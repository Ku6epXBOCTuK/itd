**Общие советы по архитектуре**

### 1. Одна плоская схема сущностей (Single Entity Type)

- Всё, что может быть у любого объекта игры, хранится в **одном** TypeScript-типе `Entity`.
- Все компоненты — **опциональные** поля (`position?`, `hp?`, `viewId?`, `isEnemy?`, `target?` и т.д.).
- Никаких отдельных классов «Башня», «Пуля», «Опыт» и никаких Union-типов.  
  **Результат**: любая система может работать с любой сущностью без приведения типов и вложенности.

### 2. Данные и визуализация — строго раздельно (View Bridge)

- В ECS-сущности **никогда** не кладутся тяжёлые объекты Three.js (Mesh, Material и т.д.).
- Хранится только лёгкий идентификатор (`viewId: "tower_lv1"`).
- Связь между сущностью и 3D-объектом делается через `WeakMap<Entity, Object3D>` внутри одной специализированной системы-моста.
- **Плюс**: можно полностью поменять рендерер (Three.js → Babylon → Pixi и т.д.), переписав только эту одну систему.

### 3. Реактивность и очистка через lifecycle-коллбэки

- Используйте `world.onEntityAdded` / `onEntityRemoved` внутри фабрик систем.
- При удалении сущности (`world.remove(entity)`) автоматически:
  - удаляется меш из Three.js,
  - убираются связанные UI-элементы.
- DeathSystem просто ставит маркер или сразу удаляет — остальное делают коллбэки.

### 4. HUD на Svelte 5 Runes + проекция координат

- Системы ECS **не рендерят** HUD сами.
- Они просто кладут ссылки на нужные сущности в реактивный `$state` массив.
- Svelte-компонент сам:
  - перебирает этот массив,
  - проецирует 3D-позицию в 2D-экранные координаты (CSS `left/top`),
  - обновляется автоматически при любом изменении данных в ECS.

### 5. Логика — как конвейер событий через маркеры

Типичная цепочка (башня → враг → опыт):

1. CombatSystem → `hp <= 0` → ставит `isDead`.
2. LootSystem → видит `isDead` → спавнит сущность «опыт» с `target: towerEntity`.
3. MoveSystem → двигает опыт к цели → при достижении добавляет xp и удаляет опыт.
4. DeathSystem → ждёт анимацию и окончательно удаляет труп.

**Ключ**: вместо прямых вызовов функций — маркеры + отдельные системы, которые их «видят».

#### Подробный пример: полный цикл смерти врага (параллельная реакция систем)

1. **Начало (HealthSystem)**  
   Система здоровья видит `hp <= 0`. Она делает ровно две вещи:
   - Добавляет компонент `isDying: true`.
   - Убирает компоненты `isEnemy` и `isTargetable`, чтобы в труп больше не стреляли.

2. **Реакция систем (Параллельное выполнение)**  
   Теперь **несколько систем одновременно** «видят» эту сущность в своих запросах:
   - **SoundSystem** (`with("isDying", "deathSound")`):  
     Видит маркер и тип звука. Проигрывает аудио и **сразу удаляет** компонент `deathSound`, чтобы звук не зациклился.

   - **VFXSystem** (`with("isDying", "position", "viewId")`):  
     Создаёт новую временную сущность «Взрыв» или «Пыль» на позиции врага.

   - **LootSystem** (`with("isDying", "experienceValue")`):  
     Создаёт сущность «Опыт» и передаёт ей значение из умершего врага. Удаляет `experienceValue` у трупа.

   - **Svelte HUD** (`$state`):  
     Видит, что сущность пропала из массива «Живые враги» (или переместилась в «Умирающие»), и автоматически запускает CSS-анимацию исчезновения полоски HP.

### 6. Ввод через Event Bus

- Кнопки в Svelte **не трогают** ECS напрямую.
- Они кидают именованное событие (`"upgrade-tower"`, payload с `entityId`).
- Система ввода в начале игрового кадра забирает события из очереди и безопасно применяет их к миру.

### Краткий чек-лист «золотых правил»

- Одна плоская `Entity` со всеми опциональными компонентами.
- Никаких Three.js-объектов внутри ECS.
- WeakMap + Bridge-система для визуализации.
- Lifecycle-коллбэки для автоматической очистки.
- Svelte Runes + проекция 3D→2D для HUD.
- Маркеры + конвейер систем вместо прямых вызовов.
- Event Bus между UI и ECS.
